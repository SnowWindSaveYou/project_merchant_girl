# 信件系统设计文档

> **跨通道叙事机制 — 打破聚落信息孤岛的第七条线路**
>
> 信件系统不是第七个独立叙事通道，而是一个**跨通道机制**：它为主线、NPC对话、订单系统提供内容输入，解决当前叙事中"远程通信缺失"的结构性问题。

---

## 一、设计动机

### 1.1 现有问题

当前游戏的六大叙事通道有一个共同限制：**所有信息传递都依赖主角亲身到场**。

根据 `narrative_guidelines.md` 第三节的规则，跨聚落信息只能通过以下渠道流动：
1. 玩家亲历
2. 同势力NPC内部传话
3. 鸣砂广播（公共信息）
4. 行商传闻（模糊二手信息）
5. 孟回（流动商人带来的消息碎片）
6. 玩家主动告知NPC

这些渠道在世界观上是合理的（废土通讯不畅），但在叙事体验上造成了一个缺陷：

> **NPC永远不能主动联系主角。**

这意味着：
- 悬念只能在"下次到访"时揭晓，无法在旅途中制造牵挂
- 聚落危机只能等主角到了才知道，缺少"紧急来信"的戏剧张力
- 角色关系发展完全依赖面对面互动，缺少书信往来的含蓄温度

### 1.2 信件系统的定位

信件系统通过一个新角色**雪冬**（邮递员）解决上述问题，同时保持"废土通讯不畅"的世界观一致性：

| 设计目标 | 解决方案 |
|---------|---------|
| 远程通信的可信来源 | 不是电子通讯，而是一个人肉投递信件的邮递员 |
| 悬念钩子的载体 | NPC可以通过信件在主角不在场时传递信息/请求/警告 |
| 新的支线叙事方式 | 主角可以接取送信委托，送达时听到收件人反应，丰富世界观 |
| 角色关系的补充维度 | 书信中的文字比面对面交谈更能展现角色内心（尤其是不善言辞的NPC） |

### 1.3 世界观合理性

废土没有邮政系统，但**有人愿意做这件事**——这本身就是一个故事。

雪冬选择当邮递员不是因为有组织指派，而是因为她相信"人跟人之间不该断了联系"。这与游戏主题一致：**修复断裂的文明纽带**。

## 二、核心角色：雪冬

> 详细角色档案见 `docs/characters/wandering_npcs.md` → 雪冬章节

### 2.1 概要

| 项目 | 设定 |
|------|------|
| **身份** | 自封邮递员 / 废土唯一的信使 |
| **性别** | 女 |
| **定位** | 流动NPC / 中前期出现 / 信件系统核心 |
| **系统ID** | `xue_dong` |
| **偏好出现位置** | 各聚落节点、中转节点（与孟回/鸣砂相同） |
| **背景设定** | 老家在塔台，但选择成为流浪邮递员 |

### 2.2 核心反差

**说话笨拙 ↔ 写信动人**

- 面对面时紧张、结巴、不会表达，常常话说到一半就低头
- 但她投递的信件（别人写的）和她自己偶尔附上的便签，文字温暖、细腻、打动人心
- 这个反差是她角色魅力的核心：她是"用行动说话的人"——说不好，就跑着去送

### 2.3 叙事功能

| 功能 | 说明 |
|------|------|
| **信件投递** | 当有NPC写了信给主角时，雪冬会出现在主角当前所在节点 |
| **送信委托** | NPC在交易所挂出送信委托，主角可以接取并送达 |
| **流浪偶遇** | 没有信件任务时，以普通流浪NPC身份随机出现 |
| **世界观载体** | 通过她投递的信件，展现NPC之间的关系网络和聚落间的联系 |

## 三、信件接收系统

### 3.1 触发机制

当剧情/事件系统触发了一封"发给主角的信"时：

```
剧情触发信件
  ↓
雪冬出现在主角当前所在节点（下一次到达该节点时）
  ↓
玩家点击雪冬 → 对话 → 领取信件
  ↓
全屏信纸界面展示信件内容
  ↓
信件存入"已读信件"存档
```

### 3.2 雪冬出现的规则

| 条件 | 说明 |
|------|------|
| **触发** | 信件队列中有未领取的信件 |
| **位置** | 主角当前所在或即将到达的节点（聚落/中转） |
| **时机** | 主角到达节点时，雪冬已经在等待（不是突然出现，而是"比你先到"） |
| **多封信** | 如果有多封未领取信件，一次对话中全部交付 |
| **对话** | 雪冬会用她笨拙的方式简单说明"谁给你的信"，然后交出信件 |

### 3.3 信件触发时机（剧情驱动）

信件不是随机生成的，每一封都是**手写的剧情内容**，由以下场景触发：

| 触发场景 | 示例 |
|---------|------|
| **主线推进后** | 完成Ch.2后，沈禾来信感谢帮忙，顺带提到温室的新问题 |
| **NPC好感达标** | 好感达到"信任"阶段后，NPC写来私人信件 |
| **聚落危机前兆** | Ch.4水泵危机前，韩策发来技术求助信 |
| **悬念埋设** | 白述来信提到在旧档案中发现了一段关于"高塔"的记载 |
| **角色关系** | 孟回来信说在某处见到了一种罕见药草，想问林砾的意见 |
| **事件链延续** | 某个随机事件的后续结果，由相关NPC通过信件告知 |

### 3.4 信件数据结构

```jsonc
{
  "id": "letter_001",
  "sender": "shen_he",                // 寄信人系统ID
  "sender_name": "沈禾",              // 显示名
  "title": "关于水泵的事",             // 信件标题（信封上）
  "content": [                         // 信件正文（支持多段）
    "林砾、陶夏：",
    "上次你们帮忙运来的水泵零件已经装好了，出水量比之前稳定多了。赵苗说今年的苗能多活两成。",
    "不过有件事想跟你们说——最近净化池的滤芯消耗比往年快了很多，不知道是水源的问题还是设备老化。下次路过塔台的时候，能不能帮我问问韩策那边有没有检测设备？",
    "路上小心。",
    "沈禾"
  ],
  "postscript": null,                 // 附言（雪冬偶尔附上的便签，可选）
  "chapter": 2,                        // 所属章节
  "trigger": {                         // 触发条件
    "required_flags": ["ch2_pump_done"],
    "min_relation": null,
    "after_trips": 2                   // 完成触发事件后经过2趟行程
  },
  "effects": {                         // 读信后的效果
    "set_flags": ["letter_001_read"],
    "unlock_hint": "shen_he_filter",   // 可选：解锁NPC对话线索
    "add_intel": null                  // 可选：添加情报条目
  },
  "audio_scene": "settlement"          // 读信时的音频场景
}
```

### 3.5 信纸UI

信件阅读采用**全屏信纸界面**，模拟手写信的质感：

| 元素 | 说明 |
|------|------|
| **背景** | 泛黄的信纸纹理，边缘微卷 |
| **字体** | 手写风格字体，每个NPC的字迹风格不同（后期可扩展） |
| **排版** | 左对齐，自然段落间距，类似真实书信格式 |
| **寄信人** | 信纸右下角的署名 |
| **附言** | 如果雪冬有便签，显示在信纸下方，用不同颜色/字体区分 |
| **操作** | 点击/滑动翻页（长信分页），点击空白区域关闭 |
| **已读标记** | 信件存入存档，可在"已收信件"中重新查看 |

## 四、送信委托系统

### 4.1 接入订单系统

送信委托通过现有的**交易所/委托系统**接入，不单独建立新系统：

```
NPC在交易所挂出"送信委托"
  ↓
主角在交易所看到委托（类型标记为"信件投递"）
  ↓
接取委托 → 信件加入货物栏
  ↓
到达目标聚落 → 找到收件人 → 交付
  ↓
收件人说几句话（展现世界观/角色关系）
  ↓
完成委托 → 获得报酬
```

### 4.2 委托数据结构

送信委托复用现有委托结构，增加信件特有字段：

```jsonc
{
  "id": "delivery_letter_001",
  "type": "letter_delivery",          // 委托类型标记
  "client": "zhao_miao",              // 委托人（寄信人）
  "client_name": "赵苗",
  "target_npc": "cheng_yuan",         // 收件人
  "target_npc_name": "程远",
  "target_settlement": "tower",       // 目标聚落
  "cargo_name": "赵苗的信",           // 在货物栏中的显示名
  "cargo_weight": 0,                  // 信件不占重量
  "reward": {                         // 报酬
    "money": 30,
    "relation_bonus": {               // 与委托人的好感
      "zhao_miao": 2
    }
  },
  "client_present": false,            // 接取时委托人是否在场
  "client_dialogue": null,            // 委托人在场时的台词（可选）
  "delivery_dialogue": [              // 送达时收件人的台词
    { "speaker": "cheng_yuan", "text": "赵苗的信？……她还好吗？" },
    { "speaker": "cheng_yuan", "text": "……谢了。替我跟她说一声，那个电路方案我改好了，下次有人过来的时候带给她。" }
  ],
  "chapter": 2,                       // 可出现的最早章节
  "trigger": {
    "required_flags": [],
    "probability": 0.3                 // 刷新概率
  }
}
```

### 4.3 委托人在场机制

部分送信委托接取时，**委托人本人会出现在交易所**：

| 场景 | 概率 | 效果 |
|------|------|------|
| 委托人不在场 | 70% | 只看到委托挂单，接取后直接出发 |
| 委托人在场 | 30% | 委托人在交易所等候，接取前会说几句话，补充背景 |

委托人在场时的对话遵循NPC对话规范（短小、自然、不像"任务发布器"）。

### 4.4 送达时的叙事设计

送信是一个**微型叙事窗口**，收件人的反应是它的核心价值：

| 设计原则 | 说明 |
|---------|------|
| **不超过3句话** | 送信不是完整对话，只是一个片段 |
| **展现关系网** | 通过反应暗示NPC之间的关系（"她还好吗？"） |
| **世界观点缀** | 通过信件内容间接传递聚落间的信息流动 |
| **偶有意外** | 极少数委托送达后会触发后续事件（如收件人拜托主角带回信） |

### 4.5 送信委托的内容类型

| 类型 | 比例 | 示例 |
|------|------|------|
| **日常联络** | 40% | 赵苗给程远的种植笔记交流、苏墨给白述的档案整理报告 |
| **物资请求** | 25% | 韩策给沈禾的水泵检测报告及零件清单 |
| **个人情感** | 20% | 匿名信件（实际来源可推测）、NPC间的私人通信 |
| **情报/线索** | 15% | 带有世界观信息增量的信件，送达后解锁新情报条目 |

## 五、跨通道整合

### 5.1 信件系统与现有通道的关系

信件系统不替代任何现有通道，而是为它们**提供输入和衔接**：

```
                    ┌─────────────────┐
                    │    信件系统      │
                    │  （跨通道机制）   │
                    └──────┬──────────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │ 主线对话  │    │ NPC对话  │    │ 订单系统  │
    │          │    │          │    │          │
    │ 信件作为  │    │ 送达反应  │    │ 送信委托  │
    │ 悬念钩子  │    │ 丰富关系  │    │ 支线叙事  │
    └──────────┘    └──────────┘    └──────────┘
```

### 5.2 各通道的具体整合方式

#### 信件 → 主线对话

信件为主线事件**提前埋线**，制造"下一站必须去"的牵引感：

```
Ch.3 主线完成后
  ↓ 2趟行程后
韩策来信："那个信号源的数据……有些不对。你们下次来塔台，我有东西给你们看。"
  ↓
玩家产生"想去塔台看看"的动力（悬念钩子）
  ↓
到达塔台 → 触发 Ch.4 前置主线对话
```

#### 信件 → NPC 对话

信件中提到的话题可以在**下次见面时的NPC对话中被引用**：

```
读了沈禾的信（提到滤芯消耗加快）
  ↓
下次在温室见到沈禾
  ↓
沈禾对话选项中多出："你说的滤芯的事……"
  ↓
展开详细讨论 → 可能触发新委托
```

#### 信件 → 订单系统

见第四节"送信委托系统"。

#### 信件 → 篝火对话

收到重要信件后，**篝火对话池可能解锁"讨论来信"的条目**：

```
读了韩策的警告信
  ↓
篝火对话池新增：
陶夏："韩策那封信你看了吗？他写的'数据不对'是什么意思？"
林砾："……不确定。但韩策不是会大惊小怪的人。"
```

#### 信件 → 驾驶闲聊

收到信件后，驾驶途中可能触发相关闲聊：

```
刚从雪冬手中接了信
  ↓
上路后，陶夏闲聊触发：
"刚才那姑娘跑那么远就为了送封信啊……挺厉害的。"
```

### 5.3 信息传递规则的扩展

信件系统为 `narrative_guidelines.md` 第三节的信息传递渠道新增第七条：

| 渠道 | 适用范围 | 限制 |
|------|---------|------|
| **雪冬信件** | NPC主动传达给主角的信息 | 仅限NPC**刻意写信**的内容，不是NPC知道的所有信息；信件内容受NPC信息权限约束 |

**关键约束**：信件不能绕过NPC的信息权限。沈禾写信也只能写她知道的事，不能在信里突然提到塔台的技术细节。

## 六、雪冬的对话设计

### 6.1 送信时的对话模式

雪冬送信时的对话遵循固定模式，简短且符合她不善言辞的性格：

```jsonc
// 典型的送信对话
{
  "steps": [
    { "speaker": "narrator", "text": "一个背着大邮包的女孩站在路边，看到你们走近，有些紧张地翻起了包。" },
    { "speaker": "xue_dong", "text": "呃……你、你们是林砾和陶夏吧？" },
    { "speaker": "taoxia", "text": "是我们！你是？" },
    { "speaker": "xue_dong", "text": "我叫雪冬。有……有信给你们。沈禾托我带的。" },
    { "speaker": "narrator", "text": "她从包里小心翼翼地抽出一个密封好的信封，双手递过来。" },
    { "speaker": "xue_dong", "text": "那……那就这样。路上小心。" }
  ]
}
```

### 6.2 偶遇时的对话模式（无信件）

没有信件任务时，雪冬以普通流浪NPC身份出现，遵循4段式深入：

| 阶段 | 对话ID | 标题 | 内容要点 |
|------|--------|------|---------|
| 初遇 | NPC_XD_001 | 路边的邮包 | 在路边整理邮包→话少紧张→陶夏主动搭话→简单自我介绍"在送信" |
| 日常 | NPC_XD_002 | 不会写的回信 | 聊起送信日常→提到有人请她帮忙代写回信但她不敢→"别人的话我写不出来" |
| 信任 | NPC_XD_003 | 塔台的邮箱 | 聊老家塔台→以前韩策让她整理旧信件→"那些信都好老了，有的字都看不清了。但你能感觉到写的人很认真" |
| 深层 | NPC_XD_004 | 没寄出的信 | 雪冬的包里有一封自己写的信，写了很多次都不满意→"有些话……写在纸上好像就能说清楚了。但真正想送到的那个人……已经收不到了" |

### 6.3 雪冬的便签

雪冬偶尔会在投递的信件旁**附上自己的小便签**，内容是她对信件/旅途/世界的私人感想：

```
"这封信沈禾写了两遍。第一遍被水泼了。她说没关系再写一遍就好。
 但我觉得她第一遍写得更好。"

"从温室走到这里用了三天。路上看到一棵开花的树。
 不知道是什么花。但很好看。我画了一下（见背面）。"
 
"韩策的字真的很难认。但我觉得他是认真写的，因为墨迹很重。"
```

便签的出现是**低频的、可选的**，作为雪冬角色塑造的彩蛋。

## 七、章节信件规划（框架）

> 以下为信件的大致规划框架，具体内容在编写对应章节剧本时细化。

### Ch.1 — 建立存在感

| 信件ID | 寄信人 | 概要 | 目的 |
|--------|--------|------|------|
| letter_ch1_01 | 沈禾 | 感谢第一次送货，提到温室的近况 | 建立信件系统、让玩家认识雪冬 |
| letter_ch1_02 | 白述 | 邀请主角下次带些旧书来，书院正在整理档案 | 白述角色塑造、书院线索引入 |

### Ch.2 — 信息增量

| 信件ID | 寄信人 | 概要 | 目的 |
|--------|--------|------|------|
| letter_ch2_01 | 韩策 | 提到收到了奇怪的信号频率数据 | **悬念钩子**：远方信号的伏笔 |
| letter_ch2_02 | 孟回 | 路上见到的异常现象（路标被挪） | 信件作为多信源之一，呼应随机事件线索 |

### Ch.3 — 悬念加深

| 信件ID | 寄信人 | 概要 | 目的 |
|--------|--------|------|------|
| letter_ch3_01 | 沈禾 | 铁锁的独家协议正在影响温室贸易 | 主线紧迫感 |
| letter_ch3_02 | 伍拾七 | 废墟那边也注意到了铁锁的动作 | 跨聚落印证，信息网络效果 |

### Ch.4 — 危机前奏

| 信件ID | 寄信人 | 概要 | 目的 |
|--------|--------|------|------|
| letter_ch4_01 | 韩策 | **紧急求助**：水泵检测数据异常 | **强悬念钩子**：直接牵引主线 |
| letter_ch4_02 | 赵苗 | 温室外围农场的水压也在下降 | 危机佐证，从另一个视角补充信息 |

### Ch.5~6 — 汇聚

## 十、设计红线

| 编号 | 红线 | 说明 |
|------|------|------|
| L-1 | **信件不传递超出NPC信息权限的内容** | 沈禾不会在信里写塔台的技术数据 |
| L-2 | **信件不替代面对面交流** | 重要的角色互动（好感推进、重大选择）仍然通过对话完成 |
| L-3 | **信件不打断行驶** | 雪冬只在节点出现，不会像随机事件一样弹出 |
| L-4 | **送信不是跑腿苦差** | 每次送达都有微型叙事回报（收件人反应），不是纯机械劳动 |
| L-5 | **雪冬不是工具人** | 她有自己的故事线、有4段式对话深入、有主题意义 |
| L-6 | **信件频率不能过高** | 纯剧情驱动，不做每日/每趟随机刷新 |
