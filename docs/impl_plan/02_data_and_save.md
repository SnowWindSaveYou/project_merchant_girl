# Phase 02 — 数据加载与存档落地

## 本阶段目标

把“配置数据”和“玩家状态”彻底分开，并把本地缓存、云端摘要、离线恢复流程跑通。

这阶段的目标不是做玩法，而是让后面所有玩法都有可靠数据基础。

---

## 本阶段只关心什么

- JSON 配置加载
- 内存缓存
- 本地缓存读写
- 云端摘要同步接口
- 数据版本号
- 离线恢复入口

## 本阶段不要做什么

- 不做复杂玩法数值
- 不做完整事件执行器
- 不做战斗逻辑

---

## 数据分类必须先定死

### 1. 配置数据

只读，不随玩家变化。

例如：

- 商品配置
- 聚落配置
- 路线配置
- 事件配置
- 战斗数值配置

### 2. 玩家状态数据

会随着玩家游玩变化。

例如：

- 信用点
- 当前货物
- 当前路线进度
- 聚落好感
- 角色关系
- 已触发旗标

## 先做数据加载器

### 最低要求

- 能读单个 JSON
- 同一路径只加载一次
- 加载失败能报清楚错误
- 支持热重载入口（开发态可选）

### 建议接口

```lua
DataLoader.load(path)
DataLoader.reload(path)
DataLoader.clear_cache()
```

### 校验要求

至少校验：

- JSON 能否解析
- 关键字段是否存在
- ID 是否重复
- 枚举值是否合法

不要等运行时报错再找问题。

---

## 本地缓存设计

本地缓存不是正式长期存档，而是：

- 断线恢复
- 崩溃恢复
- 离线补偿基准
- 网络不可用时的临时状态保存

### 文件建议

- `save/cache_main.json`

### 保存时机

- 接单后
- 到站结算后
- 关键资源变化后
- 进入后台或退出前
- 每隔固定时间自动保存一次

### 不要这样做

- 每帧写文件
- 所有按钮点击都写一次文件
- UI 临时状态也写进缓存

---

## 云端摘要设计

首版云端只同步核心字段，不同步完整大状态。

### 推荐字段

- `credits`
- `main_stage`
- `relation_linli`
- `relation_taoxia`
- `last_save_ts`
- `profile_meta`

### 为什么先只同步摘要

因为挂机网游的正式主存档最终会走云端，但首版开发不应该一上来就把完整状态拆散到几十个云字段里。

先同步摘要，能解决：

- 换设备进度不全丢
- 基础资源和阶段能恢复
- 后续再逐步扩展

---

## 本地与云端的合并规则

必须提前写清楚，不要到上线时再拍脑袋。

### 建议规则

1. 启动时先加载本地缓存
2. 如果用户已登录，再请求云端摘要
3. 比较 `last_save_ts`
4. 如果云端更新，提示是否拉取云端
5. 如果本地更新，则允许继续本地并在合适时机回推云端

### 先不做的东西

- 自动无提示双向 merge
- 字段级冲突合并
- 多端同时在线复杂覆盖策略

首版先做“时间戳更晚者优先 + 用户确认”。

---

## 离线补偿的职责边界

### `offline.lua` 只负责：

- 计算离线秒数
- 做上限截断
- 返回结果

### `core/ticker.lua` 负责：

- 根据离线秒数推进游戏状态
- 结算收益、路线、可能的到站结果

不要把离线补偿逻辑写散在存档代码里。

---

## 数据版本号

从现在开始加，不然后面一定补得痛苦。

```lua
state.meta = {
  save_version = 1,
  last_save_ts = os.time(),
}
```

配置文件也建议保留版本：

```json
{
  "version": 1,
  "events": []
}
```
