# 闲逛系统 (Stroll) 内容扩展指南

> 面向 AI 智能体和内容作者的编写手册
>
> 本文档说明闲逛场景的数据格式、扩展方法、写作规范和注意事项。

---

## 一、系统概述

### 闲逛是什么

闲逛是玩家在聚落停留时的**可选休闲活动**。消耗 1 饮用水，随机体验 2-3 个当地生活场景，获得好感、物资、旗标等回报。

### 闲逛 vs 篝火 vs 主线

| 系统 | 视角 | 内容重心 | 参与者 |
|------|------|---------|--------|
| **闲逛** | 两人一起在聚落里逛 | 与聚落居民互动、体验当地风土人情 | 陶夏 + 林砾 + 聚落 NPC |
| **篝火** | 两人私密对话 | 关系深化、回忆、情感推进 | 仅陶夏和林砾 |
| **主线** | 外部事件驱动 | 世界推进、冲突、有后果的选择 | 主角 + NPC 领袖/势力 |

**核心区别**：闲逛的重心是**聚落生活**，两人的互动是调味而非主菜。两人之间不应有深入的情感推进（那是篝火的职责），但应该体现她们作为搭档的默契和性格差异。

### 系统流程

```
玩家点击"四处逛逛"
  ↓
检查条件: 聚落节点 + 未闲逛过 + 有水 + 有可用场景
  ↓
消耗 1 饮用水
  ↓
从场景池中随机抽 2-3 个（按好感过滤）
  ↓
依次展示场景卡片 → 玩家选择 → 展示结果
  ↓
汇总页 → 返回聚落
```

## 二、数据格式

### 文件位置

```
assets/configs/stroll_scenes.json
```

### 顶层结构

以聚落 ID 为 key，每个聚落对应一个场景数组：

```jsonc
{
  "greenhouse":  [ /* 温室社区的场景 */ ],
  "bell_tower":  [ /* 钟楼书院的场景 */ ],
  "ruins_camp":  [ /* 废墟营地的场景 */ ]
  // 未来新聚落在此扩展
}
```

聚落 ID 必须与 `map_graph.json` 中的节点 ID 完全一致。

### 单个场景结构

```jsonc
{
  "id": "stroll_gh_01",          // 唯一 ID，建议格式: stroll_{聚落缩写}_{序号}
  "title": "日落浇水",            // 场景标题，显示在卡片顶部
  "text": "正文描述...",           // 场景正文（见写作规范）
  "min_goodwill": 0,              // 最低好感要求（0/20/40）
  "choices": [                    // 恰好 2 个选择
    {
      "text": "选项文本",          // 按钮文字，简短有力
      "ops": [                    // 执行的操作列表（见 ops 参考）
        "add_goodwill:3"
      ],
      "result_text": "选择后的结果描述...",  // 结果正文
      "require_goods": "metal_scrap"       // 可选：需要持有某物品才可选
    },
    { /* 第二个选择 */ }
  ]
}
```

### 好感门槛设计

| min_goodwill | 层级 | 场景定位 | 建议数量（每聚落） |
|:---:|:---:|---------|:---:|
| 0 | 外层 | 初次到访就能体验，展示聚落表面生活 | 2-3 个 |
| 20 | 中层 | 需要一定熟悉度，涉及更深入的互动 | 2-3 个 |
| 40 | 内层 | 信任关系建立后，接触聚落的秘密/核心 | 2-3 个 |

每个层级至少 2 个场景，保证随机抽取时有足够的池子。

## 三、可用 ops 参考

以下是闲逛场景中常用的操作指令：

### 经济类

| ops | 说明 | 示例 |
|-----|------|------|
| `add_goodwill:N` | 对当前聚落加好感 | `"add_goodwill:3"` |
| `add_goodwill:聚落ID:N` | 对指定聚落加好感 | `"add_goodwill:bell_tower:2"` |
| `add_credits:N` | 加/减信用币 | `"add_credits:-5"` |
| `add_goods:物品ID:N` | 获得物品 | `"add_goods:food_can:1"` |
| `add_goods:物品ID:-N` | 消耗物品（负数） | `"add_goods:metal_scrap:-1"` |
| `consume_goods:物品ID:N` | 消耗物品（语义更明确） | `"consume_goods:water:1"` |

### 关系类

| ops | 说明 | 示例 |
|-----|------|------|
| `add_relation:N` | 双角色各加约一半 | `"add_relation:2"` |
| `add_relation_linli:N` | 仅林砾加关系 | `"add_relation_linli:3"` |
| `add_relation_taoxia:N` | 仅陶夏加关系 | `"add_relation_taoxia:2"` |

### 旗标类

| ops | 说明 | 示例 |
|-----|------|------|
| `set_flag:标记ID` | 设置旗标 | `"set_flag:gh_heard_old_farmer"` |
| `clear_flag:标记ID` | 清除旗标 | `"clear_flag:temp_marker"` |

### 闲逛场景的 ops 使用建议

- **好感是核心回报**：每个选择都应有好感变动，积极选择 +2~+5，消极/失败 -1~-2
- **物品回报要克制**：偶尔给 1 个低价值物品即可，不要变成刷物资的地方
- **信用币惩罚要轻**：最多 -5~-15，不要让闲逛变成亏本行为
- **旗标用于标记体验**：记录玩家看过/做过的事，未来可被主线或篝火引用
- **关系值谨慎使用**：闲逛不是关系推进的主渠道，偶尔 +1~+2 即可

### 旗标命名规范

```
stroll 场景旗标: {聚落缩写}_{事件描述}
  gh_heard_old_farmer    ← 温室·听了老农的故事
  bt_rabbit_hat_origin   ← 钟楼·知道了兔耳帽的由来
  rc_found_car_radio     ← 废墟·发现了车载电台
  rc_secret_trade_known  ← 废墟·知道了秘密交易点
```

## 四、写作规范

### 4.1 双主角视角（最重要的规则）

**闲逛不是一个人的旅行，是两个人一起逛。** 每个场景必须体现陶夏和林砾的共同存在。

```
❌ 错误（单人视角）：
"你走进蔬菜棚，看到一排番茄。你捡起地上的番茄装好。"

✅ 正确（双主角视角）：
"两人钻进蔬菜棚。"哇，好久没见过长这么好的番茄了！"陶夏眼睛发亮。
林砾瞥了一眼角落落地的番茄，没说话——但陶夏已经猜到她在想什么。"
```

### 4.2 两人的互动模式

根据角色设定（详见 `docs/characters/protagonists.md`），闲逛中两人的互动应遵循：

| 场景类型 | 陶夏的表现 | 林砾的表现 |
|---------|-----------|-----------|
| 遇到有趣的事 | 主动凑上前、发问、感慨 | 安静观察，偶尔补充一句精准的话 |
| 需要社交的场合 | 开口谈判、聊天、打圆场 | 站在旁边，做后勤/安全工作 |
| 遇到技术/机械 | 好奇但不懂，问林砾 | 眼睛亮起来，话变多（但她自己不承认） |
| 遇到危险/冲突 | 想管闲事，冲动但不莽撞 | 默默站在能保护陶夏的位置 |
| 遇到温情的事 | 直接表达感动 | 沉默，但行动会透露内心 |

### 4.3 两人的说话风格速记

**陶夏**：
- 多用问句、感叹句、反问句
- 爱吐槽林砾："你连这个都知道？""又来了。""你偶尔说人话还挺有道理的。"
- 认真时语气会沉下来
- 代称林砾为名字或不称呼直接说

**林砾**：
- 省略号开头是标志："……嗯。""……不好说。""……没有。"
- 句子简短，不用感叹号
- 技术话题时话明显变多
- 冷幽默，一本正经说出好笑的话

```
✅ 好的对话范例：
陶夏："你是不是偷偷多拆了两个零件？"
林砾："……是备件。合理范围。"

✅ 好的对话范例：
"你搭档比你利索，"大婶打趣道。
陶夏哼了一声："她干什么都利索，特别是让人没面子的时候。"
林砾嘴角动了一下，没接话。

❌ 不好的对话：
林砾："太好了！我好开心！这里真棒！"     ← 完全不像林砾
陶夏："嗯。"                            ← 太沉默了，不像陶夏
```

### 4.4 选项文本设计

两个选项应体现**不同态度或行动方式**，而非简单的好/坏：

```
✅ 好的选项对：
"跟着一起浇水"     vs  "在旁边看看就好"         ← 参与 vs 旁观
"帮忙修理水管"     vs  "建议他们试试树脂密封"    ← 动手 vs 出主意
"尝试调解"         vs  "别管闲事"               ← 介入 vs 回避
"翻阅管理员日记"   vs  "询问能否带一本工程图纸"  ← 情感 vs 实用

❌ 不好的选项对：
"同意"  vs  "不同意"                            ← 太单薄
"帮忙"  vs  "帮更大的忙"                        ← 没有真正的选择
```

### 4.5 正文长度控制

| 部分 | 建议长度 | 说明 |
|------|---------|------|
| 场景正文 (text) | 80-200 字 | 设定场景 + 引出选择，不要太长 |
| 选项文本 | 4-10 字 | 按钮文字，一眼能读完 |
| 结果正文 (result_text) | 60-150 字 | 选择的后果 + 两人的反应 |

**不要写成小说**。闲逛是轻量级内容，玩家可能每次停留都会触发，冗长的文本会让体验变重。

### 4.6 聚落特色要鲜明

每个聚落的闲逛场景应强化该聚落的独特气质：

| 聚落 | 核心气质 | 场景应该围绕 | 典型元素 |
|------|---------|-------------|---------|
| 温室社区 | 朴素、重视水和种植 | 农业、仪式、社区生活 | 苗圃、灌溉、敬水礼、种子 |
| 钟楼书院 | 文雅、保存知识 | 书籍、教育、旧文明记忆 | 兔耳帽、传书游戏、课堂、档案 |
| 废墟营地 | 江湖气、拾荒规矩 | 翻找、交易、地盘、生存技巧 | "有主没主"、鼠叔、黑市 |
| 北穹塔台 | 理性、技术、信号 | 设备、数据、维修、观测 | 待扩展 |

**新聚落的场景必须先确定该聚落的 2-3 个核心关键词**，所有场景围绕这些关键词展开。

## 五、扩展方式

### 5.1 为现有聚落添加新场景

1. 打开 `assets/configs/stroll_scenes.json`
2. 在对应聚落的数组末尾追加新场景对象
3. 确保 `id` 不重复（按 `stroll_{聚落缩写}_{序号}` 递增）
4. 设置合适的 `min_goodwill`，保持各层级场景数量均衡
5. 构建验证

**建议每个聚落 6-10 个场景**。当前每聚落 6 个，还有扩展空间。

### 5.2 为新聚落添加场景

1. 确认新聚落的节点 ID（在 `map_graph.json` 中定义，`type: "settlement"`）
2. 在 `stroll_scenes.json` 顶层添加新 key
3. 编写至少 6 个场景（外层 2 + 中层 2 + 内层 2）
4. 无需修改任何 Lua 代码——系统会自动按聚落 ID 匹配

```jsonc
{
  "greenhouse": [ /* ... */ ],
  "bell_tower": [ /* ... */ ],
  "ruins_camp": [ /* ... */ ],
  "dome_outpost": [    // ← 新聚落：北穹塔台
    {
      "id": "stroll_do_01",
      "title": "信号塔下",
      "text": "...",
      "min_goodwill": 0,
      "choices": [ /* ... */ ]
    }
  ]
}
```

### 5.3 添加条件选项（require_goods）

如果某个选项需要持有特定物品才能选择：

```jsonc
{
  "text": "用金属片帮忙修理",
  "ops": ["add_goodwill:4", "add_goods:metal_scrap:-1"],
  "result_text": "...",
  "require_goods": "metal_scrap"
}
```

> **注意**：当前 UI 层（screen_stroll.lua）尚未实现 `require_goods` 的前端判断（按钮灰化/提示）。该字段已在数据中预留，但需要补充 UI 逻辑后才能生效。目前它只是数据标记，不会阻止玩家选择。

### 5.4 添加旗标联动

闲逛场景可以通过 `set_flag` 与其他系统联动：

**场景内 → 主线/篝火**：闲逛中设置旗标，主线或篝火对话通过 `required_flags` 检查该旗标来触发特殊内容。

```jsonc
// 闲逛场景：翻阅管理员日记
"ops": ["set_flag:bt_read_librarian_diary"]

// 后续篝火对话：引用闲逛中的经历
{
  "id": "CF_xxx",
  "required_flags": ["bt_read_librarian_diary"],
  "steps": [
    { "speaker": "taoxia", "text": "钟楼那个管理员的日记……你还记得最后一句话吗？" }
  ]
}
```

**场景间互斥/连续**：利用 `forbidden_flags` 或 `required_flags`（需在 stroll.lua 中扩展支持）来控制场景间的前后关系。

> **当前未实现**：stroll.lua 的场景过滤目前只按 `min_goodwill` 筛选，不支持按旗标过滤。如果需要场景间的前后依赖关系，需要扩展 `stroll.lua` 的 `can_start` 和场景池筛选逻辑。

## 六、注意事项

### 6.1 JSON 格式安全

**中文对话文本中的引号必须转义。** 这是之前导致整个系统无法加载的 bug 根源。

```jsonc
❌ 错误（ASCII 双引号未转义）：
"text": "老人说："你好""

✅ 正确方案 A（转义）：
"text": "老人说：\"你好\""

✅ 正确方案 B（用中文引号，无需转义）：
"text": "老人说：\u201c你好\u201d"
```

**建议**：写完后用 Python 验证：

```bash
python3 -c "import json; json.load(open('assets/configs/stroll_scenes.json')); print('OK')"
```

### 6.2 遵守叙事规范

闲逛文本必须遵守 `docs/guaji_narrative_guidelines.md` 的全部规则，重点包括：

- **世界状态权限**（第二节）：不要提及当前章节尚未揭示的信息
- **角色说话方式**（第四节）：林砾和陶夏的说话风格必须一致
- **调性红线**（第五节）：不写绝望/死亡/说教，保持温暖治愈底色
- **NPC 不要读心**：聚落 NPC 不知道主角的私事，也不知道其他聚落的内部情况

### 6.3 闲逛不推进主线

闲逛中**不应该**：
- 引入新势力或新角色（铁锁、静潮社等）
- 揭示重要世界信息（远方信号、云海高塔等）
- 触发不可逆的剧情转折
- 让 NPC 发布正式任务

闲逛中**可以**：
- 展示聚落的日常生活和文化
- 铺垫/暗示后续内容（通过旗标）
- 让玩家获得对聚落的情感认同
- 提供与聚落深化故事线（`guaji_settlements_deep.md`）呼应的细节

### 6.4 避免与篝火内容重叠

| 内容 | 放在闲逛 | 放在篝火 |
|------|---------|---------|
| 和聚落居民一起做某事 | ✅ | ❌ |
| 两人讨论刚看到的聚落景象 | ✅ 简短即可 | ❌ |
| 两人回忆过去 | ❌ | ✅ |
| 两人讨论彼此的感受 | ❌ | ✅ |
| 两人对世界观的深入讨论 | ❌ | ✅ |
| 两人之间的亲密互动 | ❌ | ⚠️ 按关系阶段 |

**经验法则**：如果去掉聚落 NPC 和场景，这段内容还成立 → 它应该是篝火而非闲逛。

### 6.5 不要让闲逛变成商店

闲逛的核心回报是**好感和叙事体验**，不是物资。

- 好感变动：每个选择 +1~+5 或 -1~-2
- 物品：偶尔给 1 个低价值物品（food_can, old_book, metal_scrap），不要给稀有物品
- 信用币：小额惩罚 -3~-15 可以有，但大额奖励不要给
- 关系值：偶尔 +1~+2，不要高于篝火的回报

### 6.6 关系阶段意识

虽然闲逛本身不按关系阶段过滤，但文本中两人的互动应该考虑关系阶段的合理性。

由于闲逛按**好感度**过滤（0/20/40），而好感度和关系值大致正相关，可以做如下映射：

| min_goodwill | 大致关系阶段 | 两人互动尺度 |
|:---:|:---:|---------|
| 0 | early | 搭档式互动，吐槽、分工、简短对话 |
| 20 | early~mid | 可以有默契的配合、小小的关心 |
| 40 | mid | 可以有无需言语的默契、偶尔温暖的细节 |

**不要在 min_goodwill: 0 的场景里写出 late 阶段的亲密感。**

## 七、未来扩展方向

### 待实现功能

| 功能 | 说明 | 优先级 |
|------|------|:---:|
| `require_goods` UI 支持 | 需要物品的选项灰化 + 提示文字 | 中 |
| 旗标过滤 | 场景支持 `required_flags` / `forbidden_flags` 过滤 | 中 |
| 冷却机制 | 同一场景在 N 次闲逛内不重复出现 | 低 |
| 北穹塔台场景 | 为第四个聚落补充闲逛内容 | 高 |
| 季节/天气变体 | 同一场景在不同天气/季节有不同描述 | 低 |
| 立绘/背景图 | 场景数据中增加 `background` 和 `portrait` 字段 | 中 |
| 场景内多步骤 | 支持类似篝火的 `steps` 多轮对话 | 低 |

### 内容扩展建议

1. **北穹塔台**是当前最需要补充的聚落，建议围绕：观测设备、信号调试、数据记录、韩策的技术执念
2. 每个聚落可以设计 1-2 个**系列场景**（通过旗标串联），形成小故事线
3. 高好感（40+）的场景可以与聚落深化故事线（`guaji_settlements_deep.md`）呼应，为专属系统做铺垫
4. 考虑加入**稀有场景**（极低抽取权重），给反复闲逛的玩家惊喜

## 八、检查清单

每次编写或修改闲逛场景后，逐项确认：

- [ ] JSON 格式正确（用 Python 验证）
