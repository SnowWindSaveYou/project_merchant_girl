# B2 委托货物系统 — 需求变更说明

> **变更版本**: v1.0
> **变更日期**: 2026-04-05
> **影响范围**: 订单系统、货舱管理、交付结算、全局 UI

---

## 一、变更背景

### 原有问题

原设计中，订单仅记录在订单簿中，货物不占用货舱空间。这导致：

- 货舱上限为 8 格的玩家可以同时接取数十格货物的订单，货舱容量形同虚设
- 订单交付时凭空扣除货物，缺乏物理一致性
- 玩家缺乏"装车管理"的策略维度

### B2 方案核心思路

**接单即装车，交付按实际**：

1. 接取订单时，货物立即装入货舱，占用实际仓位
2. 旅途中玩家可以自由使用/卖出委托货物（不强制锁定）
3. 交付时按实际剩余货物比例结算奖励
4. 通过醒目的 UI 警告帮助玩家感知"委托货物不足"

## 二、核心机制

### 2.1 接单装载

| 环节 | 原逻辑 | B2 新逻辑 |
|------|--------|----------|
| 接单条件 | 无仓位检查 | **剩余仓位 >= 订单数量**，否则按钮禁用并显示"仓位不足" |
| 接单动作 | 仅加入订单簿 | 加入订单簿 + **货物装入 cargo** + **记录 committed** |
| 货舱影响 | 无 | 实际占用仓位，可在货舱界面看到 |

```
接单流程：
  检查 cargo_free >= order.count
    ├── 不足 → 返回失败 "仓位不足"
    └── 充足 → cargo[goods_id] += count
              committed[goods_id] += count
              订单状态 = ACCEPTED
```

### 2.2 委托追踪 (committed)

新增数据字段 `state.truck.committed`，与 `state.truck.cargo` 平行：

```lua
state.truck = {
    cargo = { food_can = 5, circuit = 3 },      -- 实际持有
    committed = { food_can = 3, circuit = 2 },   -- 委托承诺
    cargo_slots = 8,                              -- 仓位上限
}
```

**关键规则**：
- `committed` 是多个订单对同一商品的累加值
- 玩家可以自由卖出/使用 committed 的货物（不锁定）
- 当 `cargo[gid] < committed[gid]` 时视为"短缺"，UI 显示警告
- 战斗/事件消耗货物时不主动保护 committed，交付时自然按实际计算

### 2.3 交付结算（三档制）

到达目的地节点时，按实际持有货物量判定交付等级：

| 交付等级 | 条件 | 奖励 | 信誉变化 |
|---------|------|------|---------|
| **全额** | 实际持有 >= 需求量 | 100% base_reward | +2 |
| **部分** | 0 < 实际 < 需求量 | (实际/需求) × base_reward | 不变 |
| **失败** | 实际 = 0 | 0 | -5 |
| **超时** | 已超过截止时间 | 0 | -5 |

**多订单竞争规则**：当多个订单需要同一商品时，按截止时间从早到晚优先分配：

```
示例：cargo.food_can = 4
  订单A: food_can × 3, 截止 10:00 → 分配 3, 全额
  订单B: food_can × 3, 截止 10:30 → 分配 1, 部分(33%)
```

### 2.4 放弃订单

放弃订单时退还货物并清除委托：

| 环节 | 动作 |
|------|------|
| 货舱 | `cargo[gid] -= min(order.count, 实际持有)` |
| 委托 | `committed[gid] -= order.count`（下限 0） |
| 信誉 | 出发地聚落 -3 |

### 2.5 订单超时

超时处理时清除委托记录，货物留在货舱不扣除：

| 环节 | 动作 |
|------|------|
| 委托 | `committed[gid] -= order.count` |
| 货舱 | 不变（货物保留在货舱） |
| 信誉 | 目的地聚落 -5 |

## 三、UI 变更总览

### 3.1 接单准备界面 (screen_prepare)

- **新增**：顶部货舱空间状态条，显示 `已用/总计` 仓位，颜色随占用率变化
- **每张订单卡**：显示 `需 N 格` 标签
  - 绿色：仓位充足
  - 红色：仓位不足
- **接取按钮**：仓位不足时禁用，文案变为"仓位不足"

### 3.2 货舱界面 (screen_cargo)

- **容量显示**：使用 CargoUtils 统一计算
- **委托信息横幅**：蓝色，显示 `共 N 件货物已委托给订单`
- **短缺警告横幅**：红色，显示 `部分委托货物不足，交付将受影响`
- **每个物品行**：
  - 持有量后追加 `(委托 N)` 后缀
  - 短缺时名称、后缀变红色
  - 短缺物品行添加红色边框
  - 使用按钮显示 `⚠` 前缀并变为 danger 样式

### 3.3 商店界面 (screen_shop)

- **新增**：头部显示货舱容量 `仓位 N/M`
- **购买按钮**：货舱满时禁用，文案变为"仓位已满"
- **持有量标签**：有委托时显示 `(委托 N)`，短缺时变红
- **卖出按钮**：卖出后将低于委托量时显示 `⚠ 卖出`，变为 danger 样式

### 3.4 行程结算界面 (screen_summary)

- **交付标签**：每张订单卡显示交付等级标签
  - `[全额]` 绿色
  - `[部分]` 橙色，附 `交付 M/N` 明细
  - `[失败]` 红色
  - `[超时]` 灰色
- **奖励金额**：使用等级对应颜色显示

### 3.5 主页面 (screen_home)

- **货舱摘要卡片**：显示 `已用/总计`，有委托物品显示 `(委N)` 后缀
- **短缺警告**：显示 `⚠ 委托货物不足`，面板添加红色边框

### 3.6 顶栏与底栏 (shell_top / shell)

- **货舱指示器**：使用 CargoUtils 统一计算
- **短缺着色**：存在短缺时货舱数字/标签变红

### 3.7 调试面板 (screen_debug)

- **货舱调整行**：有委托时显示 `(委N)` 后缀

## 四、数据结构变更

### 新增字段

```lua
state.truck.committed = {
    [goods_id] = number,  -- 该商品被订单委托的总数量
    ...
}
```

### 订单新增字段（交付后）

```lua
order.delivery_result = "full" | "partial" | "failed" | "expired"
order.delivered_count = number   -- 实际交付数量
order.partial_reward  = number   -- 实际获得奖励
```

### 存档兼容

`main.lua` 加载存档时补丁：

```lua
if not gameState.truck.committed then
    gameState.truck.committed = {}
end
```

## 五、新增模块

### cargo_utils.lua

集中管理仓位计算与委托追踪的工具模块，供所有 UI 和逻辑模块共用：

| 函数 | 用途 |
|------|------|
| `get_cargo_used(state)` | 当前货舱总占用量 |
| `get_cargo_free(state)` | 剩余空闲仓位 |
| `get_committed(state, goods_id)` | 某物品的委托数量 |
| `get_total_committed(state)` | 全部委托货物总量 |
| `is_below_committed(state, goods_id)` | 某物品是否低于委托量 |
| `has_any_shortage(state)` | 是否存在任何短缺 |
| `add_committed(state, goods_id, count)` | 增加委托记录 |
| `remove_committed(state, goods_id, count)` | 减少委托记录（下限 0） |

## 六、设计决策说明

### 为什么不锁定委托货物？

- 末世跑商的核心矛盾是"生存 vs 利益"，锁定货物会剥夺玩家在紧急情况下的选择权
- 玩家应该能够在"保命用掉燃料芯"和"保证全额交付"之间做真实取舍
- UI 的警告机制足以防止无意间消耗过多委托货物

### 为什么使用截止时间优先级分配？

- 防止后接的订单"偷走"先接订单的货物
- 最早到期的订单最可能全额交付，符合直觉
- 鼓励玩家优先完成紧迫订单

### 为什么超时不扣除货物？

- 超时意味着"没赶上交付窗口"，货物仍物理存在于货舱
- 扣除货物属于双重惩罚，对玩家不公平
- 保留货物让玩家能在下趟行程中复用
